# 仓库公开发布与分支模型

这个文档用于把当前开发态仓库整理成适合公开访问的仓库。

## 1. 分支角色

- `main`：公开展示、稳定发布、对外引用
- `develop`：日常开发集成分支

## 2. 关键原则

不要把“当前正在开发中的历史”直接当作公开主分支。

原因通常有三类：

- 历史提交里可能混入来源边界尚未整理清楚的素材
- README、流程文档、工作流和分支约定仍是内部协作口吻
- 自动发布逻辑可能还没有把开发分支和公开分支区分开

## 3. 当前仓库的实际风险

这类仓库在公开前至少要检查：

- 是否已经确认当前构建关键素材的来源与分发边界
- 是否在历史提交中留下了不准备继续公开的文件
- 是否仍把开发说明、交接说明直接放在公开首页

如果以上任意一项成立，不能只靠临时改几个文件，因为旧提交历史和默认分支语义都会影响公开效果。

## 4. 推荐做法

最稳妥的做法有两个：

1. 保留当前开发仓库为私有备份，在干净工作树基础上创建新的公开仓库
2. 或者在同一仓库中重写历史，整理掉不需要暴露的旧记录后，再把 `main` 设为公开主分支

如果你还想保留当前开发历史作为参考，请把它留在私有仓库或本地；公开仓库只保留准备给外部访问的分支与历史。

## 5. 最小公开发布清单

- 整理构建关键素材的来源说明
- 把首页 README 改成面向陌生访客的公开说明
- 明确 `main` / `develop` 分工
- 收紧 CI：明确 `develop` 只更新调试线，`main` 只更新稳定线，正式版本只由 tag 发布
- 补齐贡献、安全、行为准则等治理文件
- 选择并提交正式许可证

## 6. 推进顺序

1. 先在本地保留当前开发态分支备份
2. 确认工作树中的关键内容、素材和说明都符合你准备公开的版本
3. 决定是“新建公开仓库”还是“重写当前仓库历史”
4. 形成干净的 `main`
5. 再把 `develop` 作为后续开发分支使用

## 7. 不要做的事

- 不要把你自己都没整理清楚来源和用途的旧分支一起推到公开远端
- 不要让 `develop` 的每次 push 覆盖稳定线或正式版本下载入口
- 不要在没有许可证的情况下对外宣称仓库已经完成开源发布
