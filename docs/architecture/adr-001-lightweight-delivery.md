# ADR-001 轻量交付与测试策略

## 状态

已采纳

## 日期

2026-03-27

## 背景

当前项目面临以下现实约束：

- 用户硬盘空间紧张
- 用户没有 Android 开发基础
- 用户当前机器有`adb` 没有 Android SDK、Gradle
- 用户不希望安装 Chrome
- 用户希望看到真实的运行效果

如果继续把“本机原生 Android 构建 + 自动化浏览器验证”当作主路径，会导致早期验证成本过高，且不适合当前协作条件。

## 决策

采用“两阶段交付”：

### 阶段 1：轻量原型先行

目标：

- 验证训练流程
- 验证计时计次规则
- 验证本次训练、动作条目、组记录结构
- 让非技术用户直接看懂

形式：

- 本地 HTML 原型
- 不要求 Chrome
- 不要求 Playwright
- 不要求 Android SDK

### 阶段 2：Android 成品化

目标：

- 把已验证的训练逻辑迁移到 Android App
- 形成可安装 APK
- 后续支持历史记录、动作标准内容、离线资源

构建方式优先级：

1. 远程 CI 构建 APK
2. 最小化本地 Android Command-line Tools 构建
3. 完整 Android Studio 环境

## 理由

- 符合“先验证核心逻辑，再投入平台工程”的标准软件工程实践。
- 避免在需求未稳定前，就要求用户承担较重的环境安装成本。
- 允许项目在低依赖、低磁盘占用下继续推进。
- 保留 Android 正式交付路径，不把项目降级为单纯网页玩具。

## 后果

正面影响：

- 用户可以更早看到可理解成果。
- 对话上下文更容易控制，因为先集中在训练逻辑与数据模型。
- 项目风险更低，返工成本更低。

负面影响：

- 会出现一段“原型与 Android 正式实现并行存在”的时期。
- 需要额外维护一份轻量原型用于沟通与验收。

## 不再作为当前主路径的事项

- 安装 Chrome 仅用于自动化验证
- 在用户本机立即配置完整 Android 开发环境
- 过早引入复杂构建链和大体量依赖
