# 架构说明 v0.1

## 1. 总体架构

首版采用单应用、离线优先架构。

- 表现层：Jetpack Compose
- 应用层：训练流程编排、导航、本次训练状态管理
- 领域层：节奏模型、计次规则、本次训练聚合
- 数据层：Room 持久化训练记录，DataStore 保存偏好，JSON/本地资源提供动作内容

## 2. 为什么最终目标仍是原生 Android

- 需求高度依赖前台训练体验与本地计时稳定性。
- 本地存储、后台恢复、音频/振动提示后续都更适合原生能力。
- 首版没有跨平台刚需，先把训练模型做扎实更重要。

但在当前阶段，原生 Android 不是最先验证需求的载体。当前先用轻量原型验证逻辑，等逻辑稳定后再继续向 Android 成品收敛。

## 3. 模块边界

当前仓库先用单 `app` 模块落地，避免早期过度拆分。代码按包分层：

- `app`：应用入口与导航
- `model`：领域模型与计次规则
- `ui`：首页与后续各功能界面
- `ui/theme`：视觉主题

后续功能稳定后再考虑拆为：

- `core-model`
- `core-data`
- `feature-training`
- `feature-history`
- `feature-standards`

## 4. 关键子系统

### 4.1 训练引擎

负责：

- 管理当前动作、当前式、当前组状态
- 在训练过程中按 `2-1-2` 节奏推进阶段
- 实时显示当前阶段与秒数
- 每完成一个完整周期实时加一
- 可选播放语音提示“起”“落”
- 输出每组结果
- 聚合本次训练的总组数与总次数
- 组结束后自动切换到休息状态
- 支持休息倒计时与超时正计时

建议状态机：

- Idle
- SessionPrepared
- SetRunning
- SetCompleted
- RestRunning
- RestOvertime
- SessionCompleted

建议节奏阶段：

- Lowering
- BottomHold
- Rising

### 4.2 内容系统

负责：

- 提供六艺十式目录
- 提供标准说明、注意事项、示意图路径
- 标记内容状态

内容来源初期使用本地 JSON + 本地图片资源。后续可演进为导入式内容包。

### 4.3 历史记录系统

负责：

- 保存本次训练
- 保存动作条目
- 保存每组成绩
- 提供训练列表与训练详情查询

## 5. 技术决策

- UI：Compose，减少 XML 页面样板。
- 存储：Room，便于建模“本次训练/动作条目/组记录”关系。
- 偏好：DataStore，保存用户声音、振动、默认休息时长等设置。
- 版本管理：Git。
- 工程管理：需求、架构、路线图文档先行，后续每次对话只推进一个受控子目标。

## 6. 已知风险

- Android SDK 当前未在本机就绪，无法立即验证构建。
- 书籍内容有版权边界，不能直接把原书文本和图片无审查地打进应用。
- “书中节奏一致”必须在后续确认合法来源后再固化为正式参数。
- 用户当前不以 Chrome 和本机 Android 开发环境作为测试手段，因此验证策略必须优先支持非开发者人工验收。

## 7. 当前实施策略

当前采用：

- 轻量原型验证训练逻辑
- 文档固化需求与决策
- Android 工程骨架作为后续正式实现承接层
- 后续优先通过远程构建产出 APK 供手机安装测试

## 8. 训练页 UI 原则

训练页正式实现时，必须把信息强弱分层，不采用“所有信息同等大小堆在一页”的设计。

优先级顺序：

- 一级：当前阶段、阶段秒数、当前组次数
- 二级：当前组数、当前动作、当前式
- 三级：本次训练起始时间、说明文字、辅助信息
