# 长期协作节奏

这是一个长周期工程，不在单次对话内追求“全部做完”，而采用受控增量方式推进。

## 工作原则

- 每次对话只推进一个可封装子目标。
- 每轮都留下可追踪成果：代码、文档、决策或验证记录。
- 对话结束时总结当前状态、下一步和阻塞。
- 避免在上下文中重复加载整个项目历史，优先依赖仓库文档。

## 建议每轮节奏

1. 选定一个 Sprint 子目标。
2. 读取相关文档与代码。
3. 实施修改。
4. 做最小可行验证。
5. 写入结果与下一步。

## 上下文变长时怎么办

这不是异常，而是长周期软件工程里的正常情况。

处理方式不是继续硬扛同一个对话，而是把项目记忆外置到仓库文档：

1. 当前轮结束前，把已完成内容、未完成内容、下一步写进 `docs/process/current-status.md`
2. 需要交接时，再补一份给下一次对话直接读取的说明，见 `docs/process/handoff.md`
3. 新开对话时，不复述全部历史，只要求先读取：
   - `docs/process/start-here.md`
   - `docs/process/current-status.md`
   - `docs/process/handoff.md`
4. 之后只处理当前小目标

这就是本项目控制上下文长度的标准办法。
